# 要件定義書：Football Calendar

## 1. 開発目的

本アプリは、W杯開催にあたり、試合情報を自分で管理・閲覧できるWebアプリとして開発する。

また、就職活動におけるポートフォリオとして、単なるアプリ開発だけでなく、要件定義・設計・チーム開発・AI活用の過程も成果物として残すことを目的とする。

---

## 2. 開発背景

W杯をきっかけにサッカーを見始めるライトユーザーは、試合日程や出場選手、選手の所属クラブを把握しづらい。

そこで、W杯の試合情報をカレンダー形式で分かりやすく表示し、出場選手の所属クラブも確認できるようにすることで、W杯後にクラブリーグの試合へ興味を広げられるアプリを目指す。

選手・クラブ表示はデータ取得の制約があるため段階的に実装し、まずは試合日程・結果の充実を優先する。

---

## 3. 想定ユーザー

主な対象ユーザーは、W杯をきっかけにサッカーを見始める新規・ライトユーザーである。

---

## 4. MVPの範囲

最初に実装する最小機能は以下とする。

| 機能 | 内容 |
|---|---|
| 試合カレンダー表示 | W杯の試合日程をカレンダー形式で表示する |
| 試合詳細表示 | 対戦国、日時、会場、ステージを表示する |
| チーム別表示 | 特定の国・チームの試合だけを絞り込める |
| 選手情報表示 | 代表選手と所属クラブを表示する |
| データ保存 | 試合情報・選手情報を保存する |

### v1 実装範囲（注記）

当初の MVP には選手・クラブ表示を含むが、**v1 としてリリースした範囲は試合・チームのみ** とする。

| 機能 | v1 | 備考 |
|---|---|---|
| 試合カレンダー表示 | ✅ 実装 | |
| 試合詳細表示 | ✅ 実装 | 日時は JST で保存・表示 |
| チーム別表示 | ✅ 実装 | |
| 選手情報表示 | ❌ 未実装 | football-data.org Free プランでは squad / person API が利用できないため見送り |
| クラブ情報表示 | ❌ 未実装 | 同上 |
| データ保存 | ✅ 試合・チーム | 選手・クラブの DB モデルと API は将来用に残置 |

### v2 実装範囲

v1 の残課題に加え、試合データの更新・複数大会のカレンダー表示・過去試合結果の閲覧を v2 で取り組む。

| 機能 | v2 | 優先度 | 備考 |
|---|---|---|---|
| sync の upsert 化 | 対象 | 高 | 再 sync で kickoff・会場・status・スコア等を更新できるようにする |
| 過去の試合結果の閲覧 | 対象 | 高 | 終了した試合のスコアをカレンダー・詳細画面で確認できる。試合中は API にスコアがあれば表示も検討 |
| リーグごとのカレンダー表示 | 対象 | 高 | W杯以外（PL 等）も切り替えて表示。試合は 1 テーブルに格納し `competition_code` で区別 |
| 試合の自動更新 | 対象 | 中 | 試合結果・status を定期的に API から取り込む |
| 日時仕様の明文化 | 対象 | 中 | README 等に JST の扱いを明記。必要なら API / フロントを調整 |
| API パス `/api/v1` 統一 | 対象 | 中 | teams / matches 等を同じ prefix に揃える |
| 選手情報表示 | 未実装 | 低 | 有料 API や代替 API の調査後に判断する |
| クラブ情報表示 | 未実装 | 低 | 同上 |

選手・クラブ機能は、football-data.org Free プランでは squad / person API が利用できないため v1 では見送った。v2 ではデータ取得手段の調査を低優先度で進め、実装可否を判断する。

### v2 データ設計（概要）

試合まわりのモデル・スキーマの詳細は `docs/モデル・スキーマ設計.md` を参照する。

| 項目 | 方針 |
|---|---|
| 試合の格納 | 大会ごとに DB を分けず、`matches` テーブルに集約 |
| 大会の区別 | football-data.org の大会コード（`WC`, `PL` 等）を `competition_code` として保存 |
| status | API の値をそのまま保存。日本語ラベルはフロントで変換 |
| スコア | API の `score.fullTime` に値があるとき保存。表示はスコアが null でなければ行う |
| Team / Player / Club | v2 コアではモデル・スキーマ変更なし |

`status` ごとのスコア表示の例外（`SUSPENDED` 等）は v2 では詳細に定めず、必要になったら見直す。

## 5. v2 以降に検討する機能

以下は v2 のコア範囲（§4）には含めず、将来の拡張として扱う。

| 機能 | 理由 |
|---|---|
| iOS純正カレンダーへのコピー | 拡張機能として設計のみ考慮 |
| 通知機能 | ユーザーアカウント等の基盤整備後に検討 |
| ユーザーアカウント機能 | 現時点では不要 |

---

## 6. 非機能要件

| 項目 | 内容 |
|---|---|
| 保守性 | 機能ごとに責務を分離し、機能追加しやすい構成にする |
| 拡張性 | 将来的なカレンダー連携やリーグ追加を考慮する |
| 可読性 | READMEやコードを整理し、第三者が理解しやすくする |
| チーム開発 | GitHubでIssue・Pull Request管理を行う |
| AI活用 | 要件定義、設計、コードレビュー等にAIを活用する |
