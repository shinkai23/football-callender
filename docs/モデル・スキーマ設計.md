# モデル・スキーマ設計

## 目的
SQLAlchemy モデルと Pydantic スキーマの責務を整理する。

## SQLAlchemy モデル
### Team
- id: int
- name: str
- country: str
- short_name: str | null
- tla: str | null
- crest: str | null

### Match

**v1（実装済み）**

- id: int … football-data.org の試合 ID
- kickoff_at: datetime … sync 時に UTC から JST へ変換し、tzinfo なしで保存
- stage: str
- venue: str
- home_team_id: int … FK → teams.id
- away_team_id: int … FK → teams.id

**v2（追加予定）**

- competition_code: str … football-data.org の大会コードをそのまま保存。全試合は `matches` テーブルに格納し、大会はこのカラムで区別する
- status: str … API の `status` をそのまま保存（例: `SCHEDULED`, `IN_PLAY`, `FINISHED`）
- home_score: int | null … API の `score.fullTime.home`。未取得時は null
- away_score: int | null … API の `score.fullTime.away`。未取得時は null

**方針（v2）**

- `competition_code` は sync 時に API へ渡した大会コードをそのまま保存する。v1 既存データは `"WC"` とする
- `status` は API の値をそのまま保存する。表示用の日本語ラベルはフロントで変換する
- スコアは API の `score.fullTime` に値があるとき DB に保存する。フロントはスコアが null でなければ表示する（試合中・終了後とも想定）。未開始・延期など API が null を返す場合は null のまま
- `status` ごとのスコア表示制御（`SUSPENDED` 等の例外）は v2 では詳細に定めない。必要になったら見直す

### Player

## Pydantic スキーマ
### TeamResponse
- id: int
- name: str
- country: str
- short_name: str | null
- tla: str | null
- crest: str | null

### MatchResponse

**v1（実装済み）**

- id: int
- kickoff_at: datetime
- stage: str
- venue: str
- home_team: TeamResponse
- away_team: TeamResponse

**v2（追加予定）**

- competition_code: str
- status: str
- home_score: int | null
- away_score: int | null

### ClubResponse
- id: int
- name: str
- country: str
- league: str

### PlayerResponse
- id: int
- name: str
- position: str
- team: TeamResponse
- club: ClubResponse | null

## 設計方針
- SQLAlchemy モデルは DB 構造を表す
- Pydantic スキーマは API レスポンス形式を表す
- API では PlayerResponse に TeamResponse と ClubResponse を含める
- v1 / v2 では Create 系スキーマは作らない。データ投入は sync スクリプト経由とする
- Match は DB では `home_team_id` / `away_team_id`、API では `home_team` / `away_team` に TeamResponse をネストする
- kickoff は sync 時に JST へ変換して DB 保存する。仕様の詳細は README 等に明文化する（v2）
- v2 では sync を upsert 化し、再同期で `kickoff_at`・`venue`・`status`・スコア等を更新できるようにする
- 複数大会の切り替えは `competition_code` と API クエリで対応する。大会ごとに DB を分けない
