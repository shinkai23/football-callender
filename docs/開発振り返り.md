## 1. はじめに

Football Calendar は、2026 FIFA ワールドカップの試合情報をカレンダー形式で閲覧する Web アプリである。

v1 のスコープは試合・チームに限定する。当初 MVP に含めていた選手・クラブ表示は、外部 API の Free プラン制限により v1 では実装しない。詳細は §3.3 および §4 を参照。

- **リポジトリ**: GitHub 上で二人チーム開発
- **構成**: FastAPI バックエンド + HTML / CSS / JavaScript フロント + SQLite
- **データ源**: [football-data.org](https://www.football-data.org/) v4 API、competition: WC、season: 2026

## 2. v1 で実装したこと

### 2.1 機能

- football-data.org から W杯 2026 の試合一覧を取得し、Team / Match を SQLite に同期
- 試合 API `GET /api/v1/matches`、チーム API `GET /api/teams`
- フロント: 月次カレンダー、試合一覧、チーム絞り込み、試合詳細モーダル
- kickoff 日時: API の `utcDate` を sync 時に JST へ変換して DB 保存
- CORS 設定によりフロントから API を呼び出し可能
- 同期スクリプト: `python -m app.scripts.sync_world_cup_teams`
- バックエンドテスト pytest
- ドキュメント: README の起動手順・v1 スコープ、要件定義の v1 注記

### 2.2 設計・構成

**レイヤー構成**

```
router → service → repository → SQLite
         ↑
football_api_service … 外部 API クライアント
         ↑
sync_service … オーケストレーション
  ├── team_sync_service
  └── match_sync_service
```

**主な技術**

| 分野 | 技術 |
|------|------|
| バックエンド | Python, FastAPI, SQLAlchemy, python-dotenv |
| フロント | HTML, CSS, JavaScript。フレームワークは未使用 |
| DB | SQLite。ファイルは `backend/football_calendar.db` |
| 環境変数 | `backend/.env`。`FOOTBALL_API_KEY` 等を設定 |

**同期の流れ**

1. `get_competition_matches("WC", season=2026)` で試合 JSON を取得
2. `sync_teams_from_matches` で homeTeam / awayTeam を teams テーブルへ保存
3. `sync_matches_from_matches` で matches テーブルへ保存。kickoff は JST に変換
4. 結果を `SyncResult` で返却。teams_count、matches_count 等を含む

## 3. 直面した課題

### 3.1 外部 API 取得 → DB への落とし込み

**発生した事象**

- API レスポンスの `matches`、`homeTeam`、`awayTeam`、`utcDate`、`stage`、`venue` 等を、Team / Match モデルへマッピングする必要があった
- 試合保存時に team ID が必要なため、Team と Match の同期順・依存関係を整理する必要があった
- 同期処理を 1 箇所に集約すると見通しが悪くなった

**対応**

- `football_api_service` を sync から分離
- `team_sync_service` / `match_sync_service` に責務分割
- `sync_service` はオーケストレーションのみ担当
- 既に DB に存在する match ID は新規 insert せずスキップ。`get_match_by_id` で判定

### 3.2 日時 JST

**発生した事象**

- API は `utcDate` を UTC で返すため、日本ユーザー向けに表示・保存のタイムゾーンを決める必要があった
- JST 対応後も、既存 match はスキップする実装のため、sync だけでは kickoff が更新されない

**対応**

- `match_sync_service` で `ZoneInfo("Asia/Tokyo")` に変換し、tzinfo なし datetime として DB 保存
- フロントは `formatKickoff` で `Asia/Tokyo` を指定して表示
- 古い DB データがある場合は `backend/football_calendar.db` を削除して再 sync

### 3.3 API の有料プラン制限

**発生した事象**

- 当初 MVP には選手・クラブ表示を含めていた
- 実装・同期段階で squad `/teams/{id}` や person 系 API が 403 Forbidden となり、Free プランでは利用できないことが判明
- 要件定義時点では、エンドポイントごとのプラン制限を十分に確認できていなかった

**対応**

- v1 スコープを試合・チームのみに縮小
- README・要件定義に v1 注記を追加
- Player / Club のモデル・API コードは将来用に残置。sync とフロント表示は v1 対象外

## 4. v1 で見送ったこと

| 項目 | 理由 |
|------|------|
| 選手・クラブの外部 API 同期 | Free プランで squad / person が利用不可 |
| 選手・クラブのフロント表示 | 上記に伴い v1 対象外 |
| 試合結果の自動更新 | MVP 後の拡張 |
| API パス `/api/v1` 統一 | 動作優先のため v2 へ |

## 5. v2 / 次バージョンの実装予定

v1 の残課題に加え、試合の自動更新・リーグごとのカレンダー表示・過去試合結果の閲覧を v2 で取り組む。選手・クラブ向けのデータ取得は、有料 API や代替 API の調査を低優先度で進める。

| 優先度 | 内容 | 備考 |
|--------|------|------|
| 高 | sync の upsert 化 | 再 sync で kickoff・会場・スコア等を更新できるようにする |
| 高 | リーグごとのカレンダー表示 | W杯以外のコンペティションも切り替えて日程を表示する |
| 高 | 過去の試合結果の閲覧 | 終了した試合のスコア・結果をカレンダー・詳細画面で確認できるようにする |
| 中 | 試合の自動更新 | 試合結果やステータスを定期的に API から取り込む |
| 中 | 日時仕様の明文化 | README 等に JST の扱いを明記。必要なら API / フロントを調整 |
| 中 | API パス `/api/v1` 統一 | teams / matches 等を同じ prefix に揃える |
| 低 | football-data.org 有料 API の導入検討 | 選手・クラブ取得の可否・コストを調査し、導入するか判断する |
| 低 | 代替 API の調査・選定 | 無料または低コストで squad / person 等が取れる API を比較する |

## 7. 学び・所感

FastAPI を使った Web アプリ開発は今回が初めてだった。HTTP API を設計してフロントから呼び出す一連の流れを経験するのは初めてで、router / service / repository の役割分担を理解するのに時間がかかった。

バックエンド単体ではなく、同期スクリプト → API → フロントという全体像を頭に入れながら開発する必要があり、特に外部 API から DB への落とし込みと CORS 設定では、エラーの原因がどの層にあるのか切り分ける力が身についたと感じる。

また、要件定義の段階で API の無料範囲まで調べきれていなかったため、選手・クラブ機能を v1 から外す判断をした。初めての開発で完璧な要件設計は難しいが、動くものを出したうえでスコープを見直すことも開発の一部だと学んだ。

二人開発では git で役割分担しながら進めた。自分の担当範囲だけでなく、sync やテスト周りのコードを読むことで、Web アプリ全体の見通しが少しずつ良くなった。
